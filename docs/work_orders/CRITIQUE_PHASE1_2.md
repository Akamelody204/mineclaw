# MineClaw 阶段性复盘与自我批判 (Retrospective & Critique)

本文档基于对 Phase 1 和 Phase 2 工作内容的深入分析，旨在客观评估当前系统架构、功能实现及工程实践中的缺陷与不足。

## 1. 架构适配性与扩展性问题

### 1.1 核心状态管理 (AppState) 的耦合度
*   **现状**: `AppState` 结构体目前是一个巨大的“上帝对象”，包含了 `Config`、`SessionRepository`、`LlmProvider`、`McpServerManager`、`ToolExecutor` 和 `ToolCoordinator`。
*   **批判**: 随着 Phase 2 功能的增加，`AppState` 的构建变得极其臃肿。这种强耦合设计使得单元测试（特别是需要 Mock 部分组件时）变得困难。
*   **改进建议**: 应引入依赖注入（DI）容器模式，或者进一步拆分为更小的、独立的 Context 对象。

### 1.2 MCP 协议实现的局限性
*   **现状**: 目前仅实现了 MCP 的核心子集（stdio transport, tools/list, tools/call）。
*   **缺失**:
    *   **Resources**: 不支持 MCP 的资源（Resource）读取协议。
    *   **Prompts**: 不支持 MCP 的 Prompt 模板功能。
    *   **Notifications**: 除了初始化，缺乏通用的通知机制（如进度更新）。
*   **批判**: 虽然满足了“工具调用”这一核心需求，但作为一个通用的 MCP 宿主，功能尚不完整，限制了接入某些高级 MCP Server 的能力。

### 1.3 终端集成的交互性缺陷
*   **现状**: `terminal_server` 使用 `std::process::Command`，这是一个阻塞式的“执行-等待-返回”模型。
*   **缺陷**: 
    *   **不支持交互**: 无法处理需要用户输入的命令（如 `sudo`, `npm login`）。
    *   **无实时反馈**: 只有命令完全结束后才能看到输出，对于耗时任务（如 `cargo build`）用户体验极差。
    *   **环境隔离弱**: 所有命令都在同一个宿主环境中执行，缺乏沙箱隔离（Sandbox），存在安全风险。

## 2. 功能性缺失与未完成项

### 2.1 会话管理的持久化
*   **现状**: Phase 1 实现了内存中的 `SessionRepository`。Phase 2 中虽然扩展了功能，但依然是**纯内存存储**。
*   **批判**: 服务重启即丢失所有对话历史。这对于一个生产级应用是不可接受的。
*   **改进建议**: 必须引入数据库（SQLite/Sled）或文件系统持久化。

### 2.2 LLM 上下文管理
*   **现状**: Phase 1 报告中提到“`max_tokens` 参数目前保留但不用于 LLM 请求，留待将来实现会话上下文管理功能”。
*   **批判**: 直到 Phase 2 结束，**上下文窗口管理（Context Window Management）仍未实现**。如果对话过长，超出 LLM Token 限制，请求将直接失败。
*   **后果**: 无法支持长对话。

### 2.3 错误处理的粗糙性
*   **现状**: 虽然定义了 `Error` 枚举，但在实际 API 响应中，很多错误被简单映射为 500 Internal Server Error，缺乏具体的错误码和用户友好的错误信息。

## 3. 工程实践中的多余与不当

### 3.1 遗留代码清理不及时
*   **问题**: 项目中长期保留了 `terminal-mcp-server.js` 和 `test-mcp-server.js` 等过渡性文件，直到 Phase 2.7 才开始清理。这增加了维护者的认知负担。

### 3.2 文档与代码的同步滞后
*   **问题**: `PHASE2.md` 中列出的“管理 API”在代码中虽然有结构定义，但部分逻辑（如自动重连）并未完全落地。文档有时超前于代码实现，导致误导。

### 3.3 测试覆盖的盲区
*   **问题**: 虽然单元测试覆盖率较高，但缺乏针对 **异常流程**（如 MCP Server 崩溃、挂起、返回非法 JSON）的健壮性测试。目前的测试主要关注“快乐路径”（Happy Path）。

## 4. 总结与行动计划

MineClaw 目前是一个**功能原型（Functional Prototype）**，而非生产就绪的产品。

**优先级最高的改进项（Top 3）：**
1.  **持久化**: 实现会话数据的磁盘存储。
2.  **上下文管理**: 实现 Token 计数与历史消息截断/总结机制。
3.  **交互式终端**: 研究 PTY 集成，支持实时流式输出。

**多余/应废弃的工作：**
*   停止维护基于 Node.js 的 MCP Server 原型。
*   精简过于复杂的 `AppState` 初始化逻辑。
