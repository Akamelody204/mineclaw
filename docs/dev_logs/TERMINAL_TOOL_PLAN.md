# MineClaw 终端工具 (Terminal Tool) 开发计划

## 1. 项目概述
基于对 Zed 编辑器终端工具实现代码的分析，本项目旨在为 MineClaw 开发一套安全、可靠、可控的终端命令执行工具。该工具将作为 Agent 的核心能力之一，允许其在受限环境下执行系统命令，并实时获取反馈。

## 2. 核心分析总结 (借鉴 Zed 设计)
- **多层权限验证**：硬编码黑名单 + 用户自定义规则 (Regex) + 交互式确认。
- **异步任务管理**：使用 `Tokio` 异步运行时处理命令执行、超时和取消。
- **输出截断与安全**：对命令输出进行大小限制，并确保截断在字符/行边界，防止 Token 爆炸。
- **环境隔离**：严格校验工作目录 (CWD) 并进行路径规范化，防止路径遍历攻击。

## 3. 开发阶段规划

### 第一阶段：基础执行与核心架构 (Phase 1)
**目标**：实现最基础的命令执行能力，建立工具接口。
- [ ] **工具接口定义**：在 `src` 中定义 `TerminalTool` 结构体及 `Tool` trait 实现。
- [ ] **基础执行逻辑**：封装 `tokio::process`，支持基本命令的执行。
- [ ] **工作目录校验**：实现基础的 `cwd` 校验逻辑，确保命令在指定范围内执行。
- [ ] **超时机制**：为命令执行添加简单的 `timeout` 控制。

### 第 1.5 阶段：漏洞整改与安全加固 (Phase 1.5 - Hotfix)
**目标**：修复 Phase 1 中暴露的致命路径漏洞、注入风险及性能缺陷。
- [ ] **修复 Windows 路径 UNC 校验漏洞**：统一路径格式，移除 `\\?\` 前缀或改用更鲁棒的路径比较方式，确保 Windows 环境下 `allowed_workspaces` 校验生效。
- [ ] **重构参数处理逻辑**：彻底弃用手动拼接命令字符串。改用 `tokio::process::Command` 的 `arg()`/`args()` 接口，利用操作系统底层机制安全地传递参数，彻底杜绝注入风险。
- [ ] **正则表达式预编译优化**：引入 `once_cell` 或 `lazy_static` 预编译黑名单正则，或在工具初始化阶段完成编译，避免高频调用时的性能损耗。

### 第二阶段：安全与权限管理 (Phase 2)
**目标**：引入权限控制机制，确保安全性。
- [ ] **硬编码规则系统**：实现类似 Zed 的 `HARDCODED_SECURITY_RULES`，禁止 `rm -rf /` 等危险操作。
- [ ] **正则表达式匹配**：引入 `regex` 库，支持通过配置文件定义 `always_allow`、`always_deny` 和 `always_confirm` 规则。
- [ ] **路径规范化防护**：实现路径规范化算法，拦截带有 `..` 的路径遍历尝试。
- [ ] **配置集成**：将权限规则整合进项目的 `config` 系统。
- [ ] **交互式确认机制 (Step 5)**：实现暂停与恢复逻辑。当命令触发 `always_confirm` 规则时，工具应暂停执行并挂起任务，直到接收到外部（CLI/API）的明确确认信号，完成权限系统的核心闭环。

### 第 2.5 阶段：安全深度加固与性能调优 (Phase 2.5 - Optimization)
**目标**：消除遗留的微小缺陷，提升极端场景下的安全性和性能。
- [ ] **消除正则现场编译回退**：重构代码逻辑，确保 `compiled_` 规则字段在生产环境下永不为空，彻底移除测试用的现场编译逻辑，保障高频调用的极低延迟。
- [ ] **精细化 Tokenize 校验**：针对 `full_command` 校验逻辑，引入基于 Shell 语法的 Tokenizer。防止攻击者利用分号 (`;`)、管道 (`|`) 或 逻辑运算符 (`&&`/`||`) 绕过基础的命令字符串校验，实现指令级别的深度内容审查。

### 第三阶段：输出管理与容错 (Phase 3)
**目标**：优化输出处理，提升健壮性。
- [ ] **输出大小限制**：实现字节级别的输出限制（如默认 1MB），防止 LLM 上下文溢出。
- [ ] **智能截断算法**：确保截断后的输出仍是合法的 UTF-8 字符串，且尽可能保留完整行。
- [ ] **退出码处理**：完善对不同 `ExitStatus` 的处理，向 Agent 返回清晰的错误信息。
- [ ] **并发控制**：防止单个 Agent 启动过多的僵尸进程。

### 第 3.5 阶段：并发与截断策略优化 (Phase 3.5 - Refinement)
**目标**：解决 Phase 3 中的性能微调与输出策略局限性。
- [ ] **并发竞争压力测试与优化**：评估 `compare_exchange` 在高并发下的 CPU 表现，确保无锁实现在竞争激烈时不会造成不必要的 CPU 尖峰。
- [ ] **双向截断策略 (Head/Tail)**：重构 `truncate_output` 使其支持“保留末尾”模式。当执行失败或输出报错（stderr）时，优先返回命令输出的最后部分，确保 LLM 能够看到最关键的错误堆栈。

### 第四阶段：高级集成与增强 (Phase 4)
**目标**：深度集成到 MineClaw 的多 Agent 协作系统。
- [ ] **状态跟踪**：记录命令执行历史，供上下文管理 Agent (Context Manager) 总结和分析。
- [ ] **跨平台适配**：针对 Windows (PowerShell/CMD) 和 Unix (Bash/Zsh) 进行 Shell 环境适配。

### 第 EX1 阶段：交互式超时与长时任务管理 (Phase EX1 - Long-running Tasks)
**目标**：将超时处理从“简单强杀”升级为“交互式监控”，支持长耗时任务的断点续航。
- [ ] **异步输出监听重构**：重构 `call` 逻辑，从 `wait_with_output` 转向基于 `tokio::io` 的异步双流（stdout/stderr）非阻塞监听。
- [ ] **实时画面快照 (Snapshot)**：实现超时后的“当前画面”获取功能。即使命令未执行完，也能返回已捕获的缓冲区内容。
- [ ] **增强型任务生命周期管理**：引入 `ActiveProcessRegistry`，支持通过 PID 或任务 ID 关联已挂起的长时进程。
- [ ] **交互式续航逻辑**：支持 Agent 发出 `continue` 指令。当发生超时且无持续输出时，询问模型是否继续等待，或将其转入后台运行（标记为执行中）。

### 第 EX2 阶段：交互式分页器防护 (Phase EX2 - Pager Protection)
**目标**：彻底消除 `less` 等交互式工具导致的进程挂起风险，引导 Agent 使用非交互式替代方案。
- [ ] **环境抑制注入**：在所有终端进程启动前，强制注入 `PAGER=cat`、`MANPAGER=cat`、`GIT_PAGER=cat` 等环境变量，使主流工具自动放弃进入交互分页模式。
- [ ] **指令级拦截逻辑**：利用 Phase 2.5 的 Tokenizer，在预检阶段识别并拦截 `less` 及其变体命令。
- [ ] **智能引导机制**：当拦截发生时，不直接抛出系统错误，而是返回定制化的“MineClaw 助手提示”，告知 Agent 终端环境的非交互特性，并建议使用 `cat`、`head`、`tail` 或 `grep`。
- [ ] **管道流容错**：专门处理 `| less` 场景。确保在此类复合命令下，系统能稳定报错并给出管道优化建议，防止因 Write-Output 无效调用导致的终端状态异常。

## 4. 技术栈
- **语言**：Rust
- **异步框架**：Tokio
- **配置解析**：Serde / TOML
- **正则库**：Regex
- **测试框架**：Tarpaulin (用于覆盖率检测)

## 5. 进度评估准则
- **功能完备性**：是否能成功执行命令并返回结果。
- **安全性**：能否拦截已知的注入和路径遍历攻击。
- **资源控制**：在输出过大或执行超时时，能否正常终止并清理资源。
- **文档一致性**：代码实现是否符合本计划中的安全设计标准。
