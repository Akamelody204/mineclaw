```# Phase EX1: 交互式超时与长时任务管理 (Long-running Tasks) 开发日志

## 1. 任务背景
在 Phase 1-4 奠定的安全、并发受控、环境感知的终端工具体系之上，Phase EX1 旨在解决长耗时任务的处理局限性。目前的超时机制采用的是“简单强杀”策略，虽然保障了资源安全，但在面对编译、大规模数据索引或复杂脚本执行等合法长时任务时，无法提供灵活的控制。本阶段的核心目标是将超时处理升级为“交互式监控”，支持任务的断点续航与状态保持。

## 2. 开发进度记录

### Step 1: 异步输出监听重构 (2026-03-15)
- **目标**：重构 `call` 逻辑，从 `wait_with_output` 转向基于 `tokio::io` 的异步双流非阻塞监听，为实时快照打下基础。
- **涉及文件**：
    - `src/tools/terminal.rs`
- **修改内容**：
    - **流监听重构**：弃用 `child.wait_with_output()`，改为手动获取 `stdout` 和 `stderr` 句柄。
    - **异步循环**：使用 `tokio::select!` 宏实现对输出流的实时并发读取。
    - **缓冲区管理**：在监听过程中实时填充输出缓冲区，确保超时后仍能保留已读取内容。
- **验证状态**：
    - [x] **逻辑验证**：通过 `tokio::select!` 成功实现对 `stdout` 和 `stderr` 的非阻塞并发读取。
    - [x] **稳定性验证**：短时命令（如 `echo`）的输出捕获完整性与重构前保持一致。
    - [x] **回归测试**：100+ 项全量测试（包含功能、安全、集成）全部通过，`cargo clippy` 无警告。

### Step 2: 实时画面快照 (Snapshot) 实现 (2026-03-15)
- **目标**：实现超时后的“当前画面”获取功能。即使命令未执行完，也能返回已捕获的缓冲区内容。
- **涉及文件**：
    - `src/tools/terminal.rs`
- **修改内容**：
    - **结构体扩展**：在 `RunCommandResult` 中新增 `is_timeout` 字段，明确标识执行状态。
    - **逻辑重构**：将 `call` 中的异步监听循环改为返回 `Option<ExitStatus>`。若因超时跳出循环，则标记 `is_timeout = true`。
    - **输出保留**：在超时发生后，直接利用已填充的 `stdout_buf` 和 `stderr_buf` 构建结果，不再直接抛出错误。
    - **平台适配**：重构了退出码解析逻辑，确保在“部分成功（快照）”场景下能正确提取 OS 状态。
- **验证状态**：
    - [x] **快照验证**：新增 `test_run_command_timeout_snapshot` 测试，验证了在 1s 超时下成功捕获并返回了首行输出。
    - [x] **回归测试**：全量功能与安全性测试通过，证明快照逻辑不破坏常规命令的完整性。
    - [x] **静态检查**：`cargo clippy` 无警告。

### Step 3: 任务标识与活跃进程注册基础 (2026-03-16)
- **目标**：引入 `task_id` (UUID) 并建立 `ActiveProcessRegistry` 的雏形，在任务超时时不再强杀进程，而是将其句柄保留在内存中以便后续“续航”。
- **涉及文件**：
    - `src/tools/terminal.rs`
- **修改内容**：
    - **标识符集成**：在 `RunCommandResult` 中增加 `task_id` 字段。
    - **数据结构定义**：新增 `ActiveProcess` 结构体，保存 `Child` 句柄、输出缓冲区和启动参数。
    - **注册表集成**：在 `RunCommandTool` 中增加 `processes: Mutex<HashMap<String, ActiveProcess>>`。
    - **挂起逻辑**：修改 `call` 方法，当发生超时时，不再直接执行 `taskkill`，而是将进程状态存入注册表。
- **验证状态**：
    - [x] **挂起验证**：超时强杀逻辑已移除，进程句柄已成功保留在注册表中。
    - [x] **注册验证**：`task_id` 与 `ActiveProcess` 的映射关系已在超时路径中建立。

### Step 4: 交互式续航逻辑 (2026-03-16)
- **目标**：支持通过 `task_id` 发出续航请求，实现发生超时任务的“断点续传”。
- **涉及文件**：

- **修改内容**：
    - **指令路由**：在 `call` 方法中识别特定的续航请求格式（如仅携带 `task_id` 且不包含新命令）。
    - **状态找回**：从 `processes` 注册表中根据 `task_id` 弹出（remove）对应的 `ActiveProcess`。
    - **断点续听**：复用 Step 1 的异步监听循环逻辑，从现有的 `stdout_handle` / `stderr_handle` 继续读取数据，并补全 `stdout_buf` / `stderr_buf`。
    - **生命周期闭环**：任务完成后正常释放资源，再次超时则重新存入注册表。
- **验证状态**：
    - [x] **续航验证**：新增 `test_run_command_continue` 单元测试，模拟了“执行 -> 超时挂起 -> 任务 ID 续命 -> 完成并获取剩余输出”的完整链路。
    - [x] **资源验证**：续航请求会从注册表中 `remove` 任务，确保任务完成后句柄被正常释放；若再次超时，则重新插入。
    - [x] **静态检查**：修复了 `MutexGuard` 跨 `await` 导致 Future 无法 `Send` 的问题，`cargo clippy` 与全量测试通过。
